⏺ 修复总结

  问题诊断

  你的担忧是正确的——Elo 更新存在一致性问题，但根本原因比预想的更隐蔽：

  1. 事务已经存在：finish_game_with_elo 已经把两个玩家的 UPDATE
  放在同一个事务里，PostgreSQL ACID
  保证"要么全成功要么全回滚"，所以**"一方成功另一方失败"在当前代码中不会发生**。
  2. 真正的问题是脏读（stale read）：虽然写操作在事务内，但读操作在事务外：
  // 旧代码：用连接池读取，不在事务内！
  let red_user = user_repo.find_by_id(red_id).await?;   // ← 事务外
  let black_user = user_repo.find_by_id(black_id).await?; // ← 事务外
  2. 如果同一玩家同时结束两局游戏，两个事务可能读到相同的旧
  rating，导致后提交的事务覆盖前一个的更新，积分丢失。
  3. 死代码风险：UserRepository::update_rating()
  完全绕过事务，如果有人调用会造成不一致。

  SELECT ... FOR UPDATE 的作用

  当事务 A 对某玩家执行 SELECT ... FOR UPDATE 时：
  - 获取该行的排他锁
  - 事务 B 如果也要 SELECT ... FOR UPDATE 同一行，必须等事务 A 提交或回滚
  - 这消除了两个事务读到相同旧值、互相覆盖的竞态条件

  ✅ #7: parseFen 每帧调用 (性能)

  文件: chess-client/src/utils/chess.ts,
  chess-client/src/components/ChessBoard.vue

  - 在 chess.ts 中添加了 createFenCache()，这是一个基于闭包的备忘录，仅当 FEN
  字符串实际改变时才重新解析。
  - 修改了 findKing()，使其接受 string | Map<...>，这样 draw()
  就可以传递已缓存的棋子映射（在将军时消除了双重解析）。
  - 为 handleMouseMove 和 handleTouchMove 添加了 requestAnimationFrame 节流——在
  60fps 的鼠标事件下，每帧最多只重绘一次。
  - 在 onUnmounted 中取消待处理的 rAF。

  ✅ #8: WS 无速率限制 (安全)

  文件: chess-server/src/main.rs, chess-server/src/handlers/ws_handler.rs

  - 为 AppState 添加了 ws_rate_limit: RateLimitState (每用户 30 条消息/10 秒)。
  - 重构了 ws_handler 消息循环：Auth 和 Ping 消息绕过速率限制；所有游戏操作消息
  (MakeMove, JoinGame, LeaveGame, Resign, OfferDraw, RespondDraw) 都受到限制。
  - 速率受限的用户会收到 Error { message: "Rate limited. Slow down." } 响应。

  ✅ #9: 每次走棋 2 次 DB 查询 (性能)

  文件: ws_handler.rs, websocket/room.rs, websocket/manager.rs

  - 为 GameRoom 添加了 red_player_id 和 black_player_id 字段（在创建时从 DB
  记录填充）。
  - 添加了 player_color_from_db(user_id) 方法——通过检查存储的玩家 ID
  来确定颜色，无需查询 DB。
  - 添加了 opponent_player_id(color) 方法——返回对手的用户 ID，无需 DB 查询。
  - MakeMove：将 find_by_id DB 查询替换为 room.is_game_over() +
  room.player_color()（内存中）。
  - JoinGame：将 find_by_id DB 查询替换为 room.player_color_from_db() +
  room.opponent_player_id()（无 DB 查询）。

  ✅ #10: JWT 仅连接时验证 (安全)

  文件: ws_handler.rs

  - 将 authenticated_user 从 Option<(Uuid, String)> 更改为 Option<(Uuid, String,
  usize)>（添加了 exp 声明）。
  - 在每个 Ping（客户端每 30 秒发送一次）时，检查 exp < 
  now。如果过期，发送错误并断开连接。
  - 基于简单时间戳比较——无需重新解析 JWT。

  ✅ #11: 无界 pendingMessages 队列 (可靠性)

  文件: chess-client/src/api/websocket.ts

  - 添加了 MAX_PENDING = 50 常量。
  - 在 send() 中：对同一 game_id 的 make_move
  消息进行去重（只保留最新的）。当超出上限时，通过 shift() 强制限制。
  - 服务器验证防止过期的走棋被应用。

  ✅ #12: Advisory lock 碰撞风险 (正确性)

  文件: chess-server/src/db/repositories/game_repo.rs

  - 将两键 pg_advisory_xact_lock(int, int) 替换为单键
  pg_advisory_xact_lock(bigint)。
  - i64 哈希通过对所有 4 个 32 位 UUID 段进行 XOR 折叠并乘以 FNV 素数
